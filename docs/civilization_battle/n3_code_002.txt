**一、查询优化器实现思路（针对 `SELECT * FROM A JOIN B ON A.x=B.x WHERE A.y>100 GROUP BY A.z`）**

| 步骤 | 关键技术 | 目的 | 主要实现细节 |
|------|----------|------|--------------|
| 1️⃣ 基数估计（直方图） | - **单列直方图**（等宽/等深）<br>- **多列联合直方图**（可选）<br>- **选择率公式**：`sel = (range_overlap / total_range)` | 估算每个基本关系（A、B）经过过滤后的行数，以及连接后的中间结果大小。 | 1. 读取 A.y 的直方图 → 计算 `sel_A = P(A.y>100)`<br>2. 基数 `|A'| = |A| * sel_A`<br>3. 对连接列 A.x、B.x 使用单列直方图估算等值连接选择率：<br>`sel_join = Σ_i (freq_Ai * freq_Bi) / (|A'| * |B|)`<br>4. 预估连接结果基数 `|A' ⋈ B| = |A'| * |B| * sel_join` |
| 2️⃣ 代价模型（IO+CPU+Memory） | - **I/O 代价**：页读取次数 × 页读取延迟<br>- **CPU 代价**：元组比较、哈希函数、排序比较次数 × 每次操作单位CPU时间<br>- **Memory 代价**：是否需要 spill（磁盘临时文件）<br>- **总代价** = I/O + CPU + Memory（可加权） | 为每一种可能的物理算子（SeqScan、IndexScan、NLJ、HashJoin、MergeJoin、Sort、Aggregate）计算执行代价，为后续DP选择提供评价函数。 | 例如：<br>• **IndexScan on A.x**（若有索引）：<br>`IO = index_pages + data_pages_selected`<br>`CPU = |A'| * (index_lookup_cost + tuple_compare_cost)`<br>• **HashJoin**：<br>`Build 端（小关系）IO = pages_of_build`<br>`Probe 端 IO = pages_of_probe`<br>`CPU = |build| * hash_cost + |probe| * (probe_hash_cost + equality_check_cost)`<br>• **Sort**：外部归并排序代价 = 2 * (passes) * (input_pages) * IO_cost + CPU_sort_per_tuple * |input| * log₂|input| |
| 3️⃣ 动态规划枚举连接顺序 | - **状态**：`DP[S]` 表示集合 `S` 中的表已经 join 出的最优代价及其实现计划。<br>- **递推**：`DP[S] = min_{T⊂S, T≠∅} ( DP[T] + DP[S−T] + cost_join(T, S−T) )`<br>- **剪枝**：只考虑左深树（左连接）或使用 bushy‑tree DP 并加入 *interesting orders*（如已排序的列）以支持 MergeJoin。 | 在指数级别的连接顺序中高效找出总代价最小的左深（或 bushy）连接树。 | 对本查询只有两表 A、B，DP 只需评估两种顺序：<br>1. `A → B`（先过滤 A 再 join B）<br>2. `B → A`（先扫描 B 再 join A）<br>计算每种顺序下的中间结果基数（使用步骤1的估计）并对应的连接算子代价（步骤2），取最小者。 |
| 4️⃣ 物化策略（Materialization） | - **何时物化**：中间结果大小超过内存阈值、或后续操作需要重复访问（如 GroupBy 需要两次读取）。<br>- **物化形式**：临时堆表、哈希表或排序后的有序文件。<br>- **决策规则**：如果 `cost_without_materialization > cost_with_materialization + spill_cost` 则选择物化。 | 避免反复重新扫描大中间结果，同时控制临时存储开销。 | 本查询中：<br>• 过滤后的 `A'` 若远大于可用内存，则在 **HashJoin** 的 build 端选择将 `A'` 哈希表写入磁盘（ spill ）<br>• 若选择 **MergeJoin**，则需要对 A'.x 和 B.x 分别排序；若排序后的有序流仍不 fit 内存，则产生临时排序文件（物化）<br>• GroupBy 采用 **HashAggregate**：若哈希表超过内存阈值则 spill 到磁盘并进行多-pass 聚合。 |

**二、与 PostgreSQL EXPLAIN 的对比（以默认配置为例）**

假设统计信息已经收集（`ANALYZE`），PostgreSQL 的执行计划可能如下（仅示意）：

```
HashAggregate  (cost=... rows=... width=...)
  Group Key: a.z
  ->  Hash Join  (cost=... rows=... width=...)
        Hash Cond: (a.x = b.x)
        ->  Seq Scan on a  (cost=... rows=... width=...)
              Filter: (a.y > 100)
        ->  Seq Scan on b  (cost=... rows=... width=...)
```

**对应关系**：

| 优化器步骤 | PostgreSQL 表现 | 说明 |
|-----------|----------------|------|
| 基数估计 | `rows=` 字段（如 `Seq Scan on a` 显示过滤后行数） | PG 使用直方图和 MCV（最常见值）估计 `A.y>100` 的选择率以及等值连接选择率。 |
| 代价模型 | `cost=` 包含 **启动费用** + **总费用**（单位：`seq_page_cost`、`cpu_tuple_cost` 等） | PG 的代价同样分为 I/O（页读取）和 CPU（元组比较、哈希等），并在 `Hash Join` 中体现 build 端哈希表构建代价和 probe 端探测代价。 |
| 动态规划枚举 | PG 对两表只会考虑两种 join 顺序（左深树），并通过 `join_collapse_limit` 控制是否尝试 bushy 树。 | 这里因为只有两表，顺序唯一；若有更多表，PG 会用 DP（或贪心+遗传）枚举。 |
| 物化策略 | `Hash Join` 自动决定是否在磁盘上 spill（当 `work_mem` 不足时会出现 `Hash` 后出现 `batches=` 和 `peak memory usage`）。<br>`HashAggregate` 同样会出现 `batches=` 当哈希表超过 `work_mem`。 | PG 的 `work_mem` 参数直接对应我们在第4步中的内存阈值；超过时会产生临时文件（物化），并在 EXPLAIN 中体现为额外的 `batches` 和 `Disk` 使用。 |

**关键差异点**：

| 方面 | 本简易优化器 | PostgreSQL |
|------|--------------|------------|
| 统计信息 | 仅使用单列直方图（可选多列） | 直方图 + MCV + 列相关系数（extended statistics） |
| 代价细粒度 | 自行定义 IO/CPU/Memory 单位（需校准） | 内置成熟的成本参数 (`seq_page_cost`, `random_page
