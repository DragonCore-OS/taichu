**BFT‑Raft (5‑node, f = 1 Byzantine) – Reference Implementation in Python**  

The code below is a *minimal* but complete prototype that demonstrates the three requested pieces:

1. **Leader election** – extended Raft voting that discards votes from known‑Byzantine peers and tolerates network partitions.  
2. **Log replication** – a two‑phase commit (prepare / commit) that works even if the leader is Byzantine; followers only acknowledge a log entry after they have received matching *prepare* and *commit* messages from a quorum (≥ 3 nodes).  
3. **Safety sketch** – informal proof that once an entry is **committed** it is permanent, no two correct nodes can ever commit conflicting entries, and the system makes progress (liveness) as long as ≤ 1 node is Byzantine and the network eventually becomes synchronous.

The implementation is deliberately **self‑contained** (no external libraries) and ships with a small test‑suite that:

* spins up 5 node processes in separate threads,
* injects a Byzantine node that may send arbitrary votes / log messages,
* creates network partitions by dropping messages,
* checks the safety and liveness properties after a series of client requests.

## 1. Design Overview

| Concept | Raft baseline | BFT‑Raft change |
|---------|---------------|-----------------|
| **Quorum size** | `⌊N/2⌋+1` (majority) | `⌊(N+f)/2⌋+1 = ⌊(5+1)/2⌋+1 = 4` → we need **≥ 4** matching messages to tolerate 1 Byzantine. In practice we use **3** for the *prepare* phase (because a Byzantine leader can at most forge 1 vote) and **4** for the *commit* phase (to guarantee overlap with any other quorum). |
| **Vote validity** | Any vote with higher term is accepted. | Votes are signed (here we simulate signatures with a node‑id) and are ignored if they come from a node currently suspected of being Byzantine or if the term is stale. |
| **Log entry state** | `uncommitted → committed` (single step). | Two‑step: `uncommitted → prepared → committed`. An entry becomes *prepared* when a quorum of **prepare** acks is received; it becomes *committed* when a quorum of **commit** acks is received. |
| **Leader responsibilities** | AppendEntries RPC, heartbeats. | Leader sends **Prepare** (phase‑1) then, after collecting enough prepares, broadcasts **Commit** (phase‑2). Followers only apply an entry after the commit phase. |
| **Byzantine detection** | None. | Each node keeps a `suspect` set; if a node receives a *conflicting* Prepare/Commit for the same index/term from the same sender, it marks that sender as Byzantine and ignores further messages from it. |

The algorithm is **safe** as long as at most one node is Byzantine because any two quorums of size 4 intersect in at least 3 correct nodes, guaranteeing overlap of prepare/commit sets.
